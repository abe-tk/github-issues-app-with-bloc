## 概要

計画に基づいて実装を行い、3つの独立したレビュワーエージェント（architecture-reviewer / standards-reviewer / test-reviewer）による観点別レビューを経て品質を担保するワークフロー。

実装者（メインエージェント）とレビュワー（サブエージェント）は別コンテキストで動作するため、自己評価の甘さを構造的に回避する。

## 手順

### 1. 計画の確認

引数として渡された計画の内容を確認する。

- 計画ファイルのパスが指定された場合は読み込む
- plan-and-review で作成された計画がコンテキストにある場合はそれを使用する
- 計画がない場合はユーザーに確認する

計画の内容をユーザーに提示する:
- 実装対象のファイル数
- 実装順序の概要
- 想定される作業量

### 2. 実装

計画に記載された順序に従い、以下のスキルを参照しながら実装する。

**参照するスキル:**
- `layered-architecture` — レイヤー構成、依存方向、ディレクトリ構成
- `bloc` — BLoC/Cubit、Event/State の実装パターン
- `testing` — テストの実装

**実装の進め方:**
- 計画のファイル構成に沿って、下位レイヤーから上位レイヤーへ順に実装する
  - models → data_providers → repositories → bloc → view
- 各ファイルの実装後、対応するテストも作成する
- 実装中に計画との差異が生じた場合は、ユーザーに確認する

### 3. 観点別レビュー

実装完了後、Agent ツールで3つのレビュワーエージェントを**並列**で呼び出す。

各レビュワーには、変更したファイルの一覧をプロンプトで渡す。3つのレビュワーは互いに独立した観点でレビューするため、並列実行で問題ない。

**並列で呼び出すエージェント:**

#### architecture-reviewer
アーキテクチャ整合性のレビュー。レイヤー構成・依存方向・責務分離・BlocProvider 構成を検証する。

#### standards-reviewer
コーディング規約の準拠レビュー。BLoC命名規約・ファイル構成・import 順序を検証する。

#### test-reviewer
テスト品質のレビュー。テスト構成・モック戦略・カバレッジを検証する。

**プロンプトに含める内容:**
- 変更したファイルの一覧（実装ファイル + テストファイル）
- 計画の概要（何を実装したか）

### 4. レビュー結果の統合と提示

3つのレビュワーの結果を統合し、ユーザーに提示する。

**提示フォーマット:**
```
## レビュー結果サマリー

| レビュワー | 判定 | 指摘数（高/中/低） |
|:---|:---|:---|
| architecture-reviewer | APPROVED / NEEDS_REVISION | x / x / x |
| standards-reviewer | APPROVED / NEEDS_REVISION | x / x / x |
| test-reviewer | APPROVED / NEEDS_REVISION | x / x / x |

### 要対応の指摘（重要度: 高）
- （統合した指摘一覧）

### 推奨対応の指摘（重要度: 中）
- （統合した指摘一覧）

### 参考（重要度: 低）
- （統合した指摘一覧）
```

### 5. 修正サイクル

NEEDS_REVISION の指摘がある場合、修正して再レビューする。

**ルール:**
- 修正サイクルは**最大3回**まで
- 重要度「高」の指摘は必ず対応する
- 重要度「中」はユーザーと相談して対応を決める
- 重要度「低」は対応任意（ユーザー判断）
- 再レビューは、指摘があったレビュワーのみ再実行する

### 6. 完了報告

全レビュワーが APPROVED になったら、ユーザーに完了を報告する。

**報告内容:**
- 実装したファイルの一覧
- レビュー結果サマリー（最終）
- 修正サイクルの回数
- 残存懸念があれば記載
- 次のステップの案内（コミット、PR 作成など）
